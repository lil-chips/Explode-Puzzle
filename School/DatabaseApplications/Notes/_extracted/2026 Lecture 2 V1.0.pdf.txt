Disk Access! ØFirst rule of database performance: Disk access is the most expensive thing databases do! ØAccessing data in memory can be 10-100ns ØAccessing data on disk can be up to 10s of msØThat’s 5-6 orders of magnitude difference! ØEven solid-state drives are 10s-100s of μs (1000x slower) ØUnfortunately, disk IO is usually unavoidable ØUsually the data simply doesn’t fit into memoryØThe data needs to be persistent for when the DB is shut down, or when the server crashes, etc. ØDBs work very hard to minimise the amount of disk IO.

Minimising Disk Access! ØFor mathematically inclined:Given that an index record contains only the indexed field and a pointer to the original record, it will be smaller than the multi-field record that it points to. So the index itself requires fewer disk blocks than the original table, which therefore requires fewer block accesses to iterate through. The schema for an index on theSerialNumfield is outlined below;
Note: Pointers in MySQL are 2, 3, 4 or 5 bytes in length depending on the size of the table.Given our sample database ofr = 7,000,000records with an index record length ofR = 8bytes and using the default block sizeB = 1,024bytes. The blocking factor of the index would bebfr = (B/R) = 1024/8 = 128records per disk block. The total number of blocks required to hold the index isN = (r/bfr) = 7000000/128 = 54688blocks.Now a search using theSerialNumfield can utilize the index to increase performance. This allows for a binary search of the index with an average oflog2 54688 = 15.73 = 16block accesses. To find the address of the actual record, which requires a further block access to read, bringing the total to16 + 1 = 17block accesses, a far cry from the 7,000,000 block accesses required to find aSerialNummatch in the non-indexed table.

Indexing – To be used to not to be used?ØAs you have seen during the demo, there are three possible scenarios with use of indexes:ØIndex Scan: when query optimiser determines that it is likely to be efficient to use any available index to reach out to relevant rows in the table;ØIndex-Only Scan: when the query can be answered by just only accessing the index. The basic idea is to return values directly out of each index entry instead of consulting the associated table.ØFull Table Scan: Even when an index presents, sometimes the optimiser determines that it is quicker to retrieve data directly from the table, bypassing the index.

Example 3: Full Table Scan query SELECT * FROM ontime  WHERE tailnum LIKE 'N%';
Cost-based optimizer compares the cost of executing the query with and without index. Because there is a large proportion of rows that meets this partial condition, it determines that a full table scan is quicker than index-scan. So, it ignores the index and do a full table scan.

Index Implementations – B+ TreeØA very widely used ordered index storage format ØManages a balanced tree structure ØEvery path from root to leaf is the same length ØGenerally remains efficient for selects, even with inserts and deletes occurringØCan consume significant space, since individual nodes can be up to half empty!ØIndex updates for insert and delete can be slow.ØTree structure must be updated properly ØInserts may require one or more nodes to be splitØDeletes may require one or more nodes to be merged. ØPerformance benefits on queries more than outweigh these costs!ØQueries are straightforward: exact match, partial-match, range, etc

Height of a B+ Tree•Height = Log f (N)•F – Fanout – how many <key, pointer> pairs in a disk page•N – No. of rows in the table•Not all pages are filled to the maximum – space left in nodes to accommodate future insertions. Normally, on average 2/3 of node utilisation•Use an online logarithm calculator, such as https://www.calculator.net/log-calculator.html •E.g. 7 million records•key value (integer) = 4 bytes•Pointer = 4 bytes•Block size = 4096 bytes•Load factor = 67% (or 2 third filled on average)•Fanout ~ (approx) 343•Height = Log 343 (7000000) = 2.7 à rounded up: 3 levels 38

Index Implementations – Hash IndexØA hash index consists of a collection of buckets organized in an array. A hash function maps index keys to corresponding buckets in the hash index.ØMultiple index keys may be mapped to the same hash bucket.ØIf two index keys are mapped to the same hash bucket, there is a hash collision. A large number of hash collisions can have a performance impact on read operations.ØThe in-memory hash index structure consists of an array of memory pointers.

Filtered Index – Mitigating Disadvantages ØFiltered indexing allows us to build an index on a portion of the table – a portion that is unlikely (or less-likely) change over time and  more queries are likely be based on.ØAn example is to build an index on older rows on a transactions table where filtering is done based on transaction date. CREATE INDEX FilteredOntimeTailnumIDX      ON ontime(tailnum)      WHERE month <=10;
This builds a filtered index on rows related to first 10 months of the ontime data, assuming no new rows are added, this portion of the table is static, hence no additional index management costs.

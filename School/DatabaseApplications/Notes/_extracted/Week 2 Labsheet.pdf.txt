1 Objective The objective of this tute/lab session is to learn basics of building an efficient database backend for an application, specifically: • analyse the requirements outlined in the problem description; • develop a conceptual model to assist you with the design of the database backend required for the system; • compute estimated table and index sizes and the estimated growth for extremely large tables; • identify and implementing efficient storage strategies for extremely large tables; • identify and implementing efficient partition strategies for extremely large tables; • write efficient queries on extremely large tables and describe the query plans that query optimiser would likely be using and explain how a cost-based query optimiser would execute such query plans; • write stored procedures in T-SQL to automate common tasks on Microsoft SQL Server. RMIT Classification: Trusted Among the advanced SQL skills, I must emphasise that you should be able to write queries using: • Nested queries • Sub-Selects • Queries using IN, EXISTS • Different types of JOINs, including self-joins • Grouping and grouped aggregation -- GROUP BY, HAVING • Set operations    3.2 Activity 2: Building an efficient database backend for an application  In your first-year database concepts courses, you developed skills in analysing user specifications for a database-driven application, design the conceptual design, convert a conceptual model into a relational database model and build a database using the final physical database design. You also develop skills in writing SQL statements to carry out various CRUD tasks.   However, while you have given special attention to good design practices such as normalisation and integrity of data, it is unlikely that sufficient attention was given to the efficiency characteristics of the underlying database design. In this activity, you will do a fully-fledged (from the user requirements to all the way to final database deployment) while focussing on the efficiency characteristics at each stage of your database design.  Problem Description  Victoria’s public transport ticketing system (MyKi) is aging and up for a replacement. The governing body – Public Transport Victoria (PTV) – has called for tenders for a replacement system. The multi-national infrastructure development company you just joined won the contract. You are now in the project team, with your database skills, and tasked to design the database back-end.  In the Phase 1 of the project, it is not expected to change the basic ticketing guidelines (such as zones, MyKi Money and MyKi Pass options, or the pricing). In this phase, you will simply replace the aging infrastructure, including the substantially slow database infrastructure. In fact, after your initial investigation on the performance issues in the current system, you found out that database infrastructure is one of the bottlenecks in the system and it slows down the operation of the whole system.

Why cards?
User stories emerged in Extreme Programming (XP)
The early XP literature talks about story cards rather than user stories. 
Why?
User stories were captured on paper cards. 
Benefits: 
1. Paper cards are cheap and easy to use. 
2. They facilitate collaboration: Every one can take a card and write down an idea. 
3. Cards can be easily grouped on the table or wall to check for consistency and 
completeness and to visualize dependencies.  
Even if your stories are stored electronically, it might be worthwhile to use paper 
cards when you write new stories. https://www.romanpichler.com/blog/10-tips-writing-good-user-stories/

Acceptance criteria (AC): 
Given – When –Then 
GIVEN some context
WHEN some action is carried out
THEN a particular set of measurable consequences should be observable
These ACs look more complex, but most 
companies use this format as it’s more precise.
Behavior-driven development
There should be only one action in the WHEN -part 
→ If you need to cover 2 or more actions, you will 
need to specify 2 or more ACs

To wrote ACs, a User Story should be small
Larger stories = hidden assumptions and ambiguity during the development
Example:  As a website user, I want to post my resume so that I can be contacted 
regarding job interviews. 
Too vague and compound story because
• A resume includes separate sections for education, prior jobs, salary history, 
publications, etc.
• Users can mark resumes as inactive
• Users can have multiple resumes
• Users can edit resumes
• Users can delete resumes
What to do with a bigger story?

Epics
• A large user story is typically referred to as an epic 
• There is no “magic size” at which we start calling a user 
story an epic
• Generally: an epic is a user story that will take more 
than one or two sprints to develop and test 
• Epics are split into smaller user stories, so that a story 
can be completed within a sprint
https://www.ibm.com/developerworks/rational/library/user-stories-product-
backlog/index.html
Is this an epic?
As a user, I want to log in to the system so that 
my information can be accessed only by me. 
Assume that the product owner clarifies that this 
simple story is intended to cover everything 
related to logging in: 
• requesting a new password, 
• changing the password, 
• ... 
It is about more than pressing a Log in button on 
one screen.
 

From an epic to a user story
As a user, I want to log in to the system so that my information can be 
accessed only by me.  
• As a new user, I want to register by creating a username and password so that the 
system can remember my personal information.
• As a registered user, I want to log in with my username and password so that I can 
trust the system.
• As a registered user, I want to change my password so that I can keep it secure or 
make it easier to remember.
• As a registered user, I want the system to warn me if my password is easy to 
guess so that my account is harder to break into.
• As a registered user, I want to be able to request a new password so that I am not 
permanently locked out if I forget it.
• As a registered user, I do not want to be sure if it was the username, password, or 
both that was wrong when my login attempt fails so that someone trying to 
impersonate me will have a harder time doing so.

Themes:
• groups of related stories
• contribute to a common goal or are related 
in some obvious way, such as all focusing 
on a single customer. 
• stories in a theme do not need to 
encapsulate a specific work flow or be 
delivered together.
• some stories in a theme may be dependent 
on one another, but are independent 
enough to be delivered separately and still 
provide some business value 
Epics:
• groups of related stories
• comprise a complete work flow 
for a user 
• the stories that comprise an epic 
may be completed 
independently, but their 
business value isn't realised 
until the entire epic is complete 
• it rarely makes sense to deliver 
an epic until all of the underlying 
stories are complete 
What makes a product backlog good / bad?
The product backlog should be available for anyone to see.
The backlog must be the single source of requests.                                              
It’s important for the source of requests to be centralized because the entire list 
needs to be ordered.
The product backlog is a living artifact and is never complete.                          
It’s very normal to identify new items, remove items that are no longer 
considered valuable and to morph existing items.                                             
The product backlog lives for as long as the product lives.
Using the backlog as a dumping ground of ideas to appease stakeholders.
Having more than 1 backlog 
The Scrum guide suggests you spend no more than 10% of your team’s time 
keeping the backlog in working order. 
Grooming your backlog ensures that it always exhibits the healthy 
characteristics that make the backlog a pleasure to work with. If you don’t groom 
the backlog you’ll run into difficulty planning and delivering backlog items. 

Is our Product Backlog correct?
✓ Are the PBI IDs unique? Do the IDs make sense? 
✓ Are the PBI written as reasonable user stories? (reasonable semantics, 
correct format As a… I want to … so that …)
✓ Is the product backlog sorted by the priorities of the PBIs?
✓ Are the priorities correct wrt. notes made during the initial requirements 
elicitation interviews with the stakeholders?
✓ Are the Owners reasonable wrt. the PBIs?
✓ Are the Efforts reasonable wrt. the PBIs?
✓ Are the DoDs reasonable wrt. the PBIs?

vs. 
Acceptance 
Criteria
DoD is intended to be 
universally applicable  
             (quality!)
Acceptance Criteria are specific 
for each feature  
                 (functionality!)
DoD should NOT include any 
functionality!
Definition of Done (DoD)
• List of activities that add verifiable/demonstrable value to the product  
• DoD is orthogonal to user acceptance criteria (functional acceptance) 
for a feature: It is a comprehensive checklist of necessary, value-added 
activities that assert the quality of a feature and not the functionality of 
that feature.  
• Ensures that only truly done features are delivered, not only in terms of 
functionality but in terms of quality as well. 
 
• The exit-criteria to determine whether a product backlog item is complete. 
• In many cases the DoD requires that all regression tests should be successful. 
• The definition of "done" may vary from one Scrum team to another, but must be 
consistent within one team.

Definition of Done
Goals:
• to build a common understanding within the Team about Quality and Completeness
• to be a checklist that User Stories (or PBIs) are checked against
• to ensure the increment shipped at the end of the Sprint has high quality and that the 
quality is well understood by all involved.
 
Example of DoD:
✓ Code is either reviewed or produced with a pair -programming method
✓ The code is covered by Unit Tests (minimum of 70% code coverage) and all tests are Green 
(means it has passed )
✓ Automated acceptance tests (based on the Acceptance Criteria of the User Story) are 
conducted and passed
✓ Integration tests of the affected areas are conducted and passed
✓ Usability Review Completed, Tested on iPhone 6 -11
✓ Regression testing
Definition of Done 
(DoD, Done Done)
• should focus on how you will ensure 
the quality of the product you develop 
– what kind of tests you are going to 
conduct, what quality criteria you will 
apply to each item
• DoD == the rules that the 
development team, Product Owner, 
and to some extent the organisation 
have agreed on when they mean a 
story is ready for delivery

Example of DoD
▪ Unit testing completed (coverage 80%)
▪ Peer reviewed 
▪ Architecture updated (if necessary)
▪ Functional testing based on ACs completed
▪ Regression testing completed 
▪ Performance testing benchmark achieved
▪ User documentation updated to reflect new story
▪ Approved by the Product Owner

DoD: Levels
There might be different DoD at various levels:
❑ DoD for a Product Backlog Item (e.g., writing code, 
tests and all necessary documentation)
❑ DoD for a Sprint (e.g., install demo system for review)
❑ DoD for a Release (e.g., writing release notes)
• Can we do this activity for each feature? If not, 
then
• Can we do this activity for each sprint? If not, then
• We have to do this activity for our release!

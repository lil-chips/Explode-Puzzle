VM overview
• Fundamental idea
– Abstract hardware of a single computer into several different execution
environments (virtual machine, or VM)
• Several components
– Host: underlying hardware system
– Virtual machine manager (VMM) or hypervisor creates and runs VMs
– Guest: VM, process provided with virtual copy of the host
• Single physical machine can run multiple operating systems
concurrently, each in its own VM

RMIT Classification: Trusted
Hypervisors
• Hypervisors should score well in three dimensions:
– Safety (resource control): hypervisor should have full control of
virtualized resources.
– Fidelity (equivalence): behaviour of a program on a virtual machine
should be identical to same program running on bare hardware.
– Efficiency: much of code in virtual machine should run without
intervention by hypervisor.

Hypervisors: Benefits and Features
• Run multiple, different OSes on a single machine
• Host system protected from VMs, VMs protected from each other
– i.e., A virus less likely to spread
– Sharing is provided though via shared file system volume, network
communication
• Freeze/suspend running VM
– Then can move or copy somewhere else and resume
– Snapshot of a given state, able to restore back to that state
– Some VMMs allow multiple snapshots per VM
– Clone by creating copy and running both original and copy

===== Page 19 =====
RMIT Classification: Trusted
Hypervisors: Benefits and Features
• Templating – create an OS + application VM, provide it to
customers, use it to create multiple instances of that combination
• Live migration – move a running VM from one host to another!
– No interruption of user access
• All those features taken together -> cloud computing
– Using APIs, programs tell cloud infrastructure (servers, networking,
storage) to create new guests, VMs, virtual desktops

===== Page 28 =====
RMIT Classification: Trusted
Why do we need these two modes?
• User mode
– No direct access to hardware
– No direct access to privileged instructions (I/O, memory management)
– A crash only affects the involved process
• Kernel mode
– Full access to hardware
– Full access to machine instructions.
– A crash is catastrophic
• Protecting OS and the system from errant users

Virtualization: what is the problem?
• Guest/VM apps run in user mode as usual
– It is fine, no problem
• Guest OS runs in kernel mode
– This is problematic
– Not safe to let guest kernel run in kernel mode, having direct access to hardware
So,
• VM needs to run in user mode
– Processes: virtual user mode
– Kernel: virtual kernel mode
– Both run in real user mode
• Hypervisor runs in kernel mode

RMIT Classification: Trusted
Virtualization: Trap-and-Emulate
• VM OS attempting a privileged instruction in user mode causes an
error  trap
• Hypervisor gains control, analyses error, executes operation as
attempted by VM  emulate
• Returns control to guest in user mode
• Most virtualization products use this technique
– However, there are performance issues (to be discussed)

Virtualization: memory management in action
• Virtualization makes memory management even harder and
confusing.
– VM OS still needs to maintain page tables for its VM, but it no longer has
direct access to memory
– Hypervisor also needs to maintain page tables for all VMs.
• You can imagine hypervisor has to work hard and get the page
table and mapping right.
– Provide the actual mapping
– Avoid conflicts among VMs

===== Page 11 =====
RMIT Classification: Trusted
Virtualization: Nested Page Tables (NPTs)
• We now have guest virtual address, guest physical address and host
physical address
• Eventually, we need to translate from guest virtual address to host
physical address
• Each VM maintains page tables to translate guest virtual address to
guest physical addresses, business as usual
• Hypervisor maintains per VM NPTs to provide mapping to host physical
address
• VM tries to change its page table  Hypervisor makes equivalent
change to NPTs
11
===== Page 12 =====
RMIT Classification: Trusted
Virtualization: NPTs in hardware
• Every time for a memory access, we need to do:
– Go through the VM page table to translate the guest virtual address to a
guest physical address
– Also go through the NPTs to finally find the host physical address
• Fortunately, CPUs implement NPTs in hardware
– hypervisor fully controls the paging
– CPU accelerates the translation.

RMIT Classification: Trusted
Virtualization: Type 2 hypervisor
• Run on top of host operating systems
• Less interesting from an OS perspective
– Very little OS involvement in virtualization
– Hypervisor is simply another process, run and managed by host
• Even the host doesn’t know it is a hypervisor running guests
• Tend to have poorer overall performance because can’t take
advantage of some HW features
• But also a benefit because require no changes to host OS

===== Page 25 =====
RMIT Classification: Trusted
VM Migration
Live migration: move the virtual machine while it is still operational.
• Copy memory pages while the VM is still serving requests.
• A page may be modified after it has already been copied, mark
them as dirty.
• When most memory pages have been copied, pause very briefly
to copy the remaining dirty pages and resume the VM on the new
machine.

Virtual Appliances
• Installing new application programs may not be easy
– Many applications are dependent on numerous other applications and libraries,
which are themselves dependent on a host of other software packages, and so
on.
– There may be dependencies on particular versions of the compilers, scripting
languages, and the operating system.
• Virtual appliances: create a VM image for installing
– A software developer can carefully construct a virtual machine, load it with the
required operating system, compilers, libraries, and application code, and freeze
the entire unit, ready to run.
• Amazon’s EC2 cloud has many pre-packaged virtual appliances
available for its clients, which it offers as convenient software services
(‘‘Software as a Service’’).

OS-level virtualization
• VMs: hardware virtualization
• VMs are heavyweight, not always needed
• In some instances, what we want to do is:
– segregate applications,
– manage their performance and resource use, and
– create an easy way to start, stop, move, and manage them.
• Can do those things without hardware virtualization
• If the applications are all compiled for the same operating system, OS-
level virtualization or application containment is good enough
